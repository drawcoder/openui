# 项目上下文

## 代理阅读契约

本文档是本仓库进行 OpenSpec 工作时的项目级权威上下文。

在创建、更新或执行 OpenSpec 变更前，应先阅读本文档。

优先级顺序：

1. 人的明确指令
2. 本项目上下文
3. 已有规格
4. 活跃变更

本文档只记录稳定的项目目标、边界、架构原则和包职责。不记录短期路线图、活跃任务清单或具体功能的设计细节。

## 项目目标

OpenUI 的目标是让应用能够接收用户意图和业务数据，自动生成正确、美观、可交互的 UI。

OpenUI 不以固定数据形状、固定内容领域或固定业务场景作为前提。面对新的数据形状时，系统应优先扩展通用能力，而不是为单个样例写特例。

生成 UI 的质量目标同时包括语义正确性和视觉质量。语义正确性优先于视觉美化：UI 必须忠实表达用户意图和输入数据，不能丢数据、错配字段、误解单位、伪造不存在的信息，或用错误组件表达数据关系。视觉质量应在不牺牲正确性的前提下提升，包括清晰的信息层级、合适的组件选择、可读布局和专业视觉呈现。

用户意图指导呈现方向，数据事实约束可被真实渲染的内容。当数据不足以支持用户请求时，系统应选择最接近且诚实的表达方式，而不是编造缺失数据。

## 用户

### 主要用户：上层开发者

OpenUI 的直接用户是把生成式 UI 能力集成到自己产品里的上层开发者。他们需要：

- 定义和组合自己的组件库
- 提供用户意图、业务数据、工具和上下文
- 控制渲染边界、交互行为、状态更新和部署方式
- 用提示词、结构定义、规格、样例和评测反馈让模型更稳定地产生符合产品语义的 UI
- 在不分叉核心语言或运行时的情况下接入自定义组件、设计系统和业务规则

### 次要用户：终端用户

终端用户通过宿主应用使用 OpenUI 生成的界面。他们关心结果是否正确、清晰、美观、可操作，但通常不直接消费 OpenUI 接口。

### 生产者：大语言模型和开发代理

大语言模型和开发代理是 UI 生成链路里的生产者。OpenUI 需要为它们提供清晰的结构定义、提示词、规格、样例和评测反馈，使它们更容易生成和改进正确、美观、可渲染的 UI。

## 架构

OpenUI 的核心链路是：

```
用户意图 + 业务数据 + 选定组件库
  -> 后端生成层
  -> OpenUI Stream
  -> 前端运行时
  -> 可交互 UI
```

OpenUI 分离语言、运行时和组件库：

- 语言定义模型输出契约。
- 运行时负责解析、渲染、状态管理和动作事件。
- 组件库定义可生成组件集合、面向提示词的结构定义、描述和渲染实现。

组件是可组合的能力单元。一个组件声明自己的名称、面向提示词的描述、类型化结构和运行时渲染器。

组件库是由组件组合出来的生成边界。它定义某个应用、领域或任务中模型可以生成什么组件、使用什么结构、如何描述这些能力，以及生成出的 OpenUI Stream 如何被渲染。

上层开发者应能通过组合内置组件、自定义组件、领域组件和受约束的组件变体来构建组件库，而不需要修改核心语言或运行时。

## 部署形态

OpenUI 以可嵌入 SDK 形态交付，而不是要求使用集中式托管服务。

### 前端

前端主要通过 npm 包集成：

- 运行时渲染
- 流式解析
- 状态和动作
- 组件库
- 开发者自定义组件库

当前前端运行时主要面向 React 应用，但 OpenUI Lang 不应绑定到单一前端框架。

### 后端

后端 SDK 嵌入开发者自己的 Java / Node.js 服务。Java 和 Node.js 是主要目标环境，但架构不应假设 OpenUI 依赖某个特定后端框架、托管模型或传输协议。

后端 SDK 分两层：

1. 提示词构造

   根据选定组件库、结构定义、规则、样例和可选工具输出模型指令。

2. UI 生成

   输入用户意图、业务数据、选定组件库和可选上下文，输出 OpenUI Stream。

宿主应用负责：

- 数据获取
- 权限
- 租户
- 审计
- 持久化
- 部署
- 后端和前端之间的传输
- 业务侧动作处理

OpenUI 负责 UI 生成契约：用户意图、业务数据、组件库、结构定义、提示词和生成结果如何形成 OpenUI Stream，以及前端运行时如何消费该流并渲染 UI、发出结构化动作和状态事件。

动作事件回到宿主应用的处理器。OpenUI 可以定义事件形态和辅助能力，但业务行为归宿主应用所有。

## 包职责

`packages/react-ui` 是 OpenUI 官方参考组件包。它代表 OpenUI 对通用组件能力、交互模式和体验质量的官方参考实现。

`packages/react-ui-dsl` 是本仓库正在实现和演进的 DSL 组件库方案。它应参考 `react-ui` 的组件语义和体验质量，但当不同 API 或结构定义能让模型更清晰、更稳定、更容易正确生成 OpenUI Lang 时，可以采用不同设计。

`react-ui-dsl` 不应在手写组件 API 不适合模型生成时机械复制这些 API。它的结构设计应优先服务模型友好生成，同时保持正确的组件语义和高质量 UI 输出。

## 与 OpenSpec 的关系

`openspec/project.md` 定义项目级默认原则、边界和决策依据。

`openspec/specs/*/spec.md` 定义能力行为要求。

`openspec/changes/*` 定义某次变更的执行语境。

当 OpenSpec 工件存在歧义时，应通过本项目上下文解释。当某个变更看起来与本项目上下文冲突时，除非人明确说明，否则优先遵循本项目上下文。
